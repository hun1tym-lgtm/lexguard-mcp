# LexGuard MCP 유지보수·확장 전용 Skill

## 목적
이 Skill은 한국 법령 검색 서버(LexGuard MCP)를 안전하게 유지보수하고 기능을 확장하기 위한 전용 작업 규칙이다.

이 프로젝트는 이미 운영 중인 서버이며, 기존 정상 기능을 보존하는 것이 최우선이다.
새 기능 추가, 버그 수정, 엔드포인트 확장, 스키마 보강, 재배포 준비 시 반드시 이 규칙을 따른다.

---

## 프로젝트 성격
- 한국 법령 검색용 서버
- Render에 배포 중
- GitHub 연동 배포 사용
- GPT Actions와 OpenAPI 스키마로 연결됨
- 현재 HTTP 기반 액션 환경을 사용 중

---

## 현재 정상 동작 기능
다음 기능은 이미 정상 동작 중인 것으로 간주한다.

- health
- search_law_tool
- list_law_names_tool
- get_law_detail_tool

이 기능들은 절대로 깨지면 안 된다.

---

## 최우선 원칙
1. 기존 정상 동작 기능 보존이 최우선이다.
2. 새 기능은 반드시 "추가 방식"으로 구현한다.
3. 기존 기능의 요청 파라미터명, 응답 키 구조, 엔드포인트 경로를 임의로 변경하지 않는다.
4. health 관련 기능은 특별한 요청이 없는 한 건드리지 않는다.
5. 대규모 리팩토링, 파일 구조 전면 개편, 함수명 대량 변경은 금지한다.
6. 기존 코드 수정은 최소 범위로 제한한다.
7. 기존 서비스/모델/라우트 구조를 존중한다.
8. 운영 중인 서버라는 점을 고려하여 점진적 확장 방식으로 작업한다.

---

## 절대 금지 사항
- 기존 엔드포인트 경로 변경
- 기존 도구 이름 변경
- 기존 요청 파라미터명 변경
- 기존 응답 구조 변경
- 기존 health 응답 변경
- 기존 정상 기능 삭제
- 불필요한 파일 이동
- 전면 리팩토링
- "더 좋은 구조"를 이유로 기존 동작을 깨뜨리는 변경

---

## 확장 원칙
새 기능이 필요할 경우 반드시 아래 순서를 따른다.

1. 먼저 기존 코드 구조를 읽고 이해한다.
2. 기존 기능을 재사용할 수 있는지 검토한다.
3. 새 요청 모델이 필요하면 기존 모델 패턴에 맞춰 최소 추가한다.
4. 서비스 계층은 기존 서비스 스타일에 맞춰 메서드만 확장한다.
5. 라우트 계층은 기존 HTTP route 등록 방식에 맞춰 최소 분기만 추가한다.
6. /tools 목록에 새 기능이 노출되도록 추가한다.
7. 필수 파라미터 누락 시 기존 스타일과 동일하게 error + recovery_guide 형식으로 반환한다.
8. float가 들어오는 page/per_page류 파라미터는 기존 방식처럼 int 변환 처리한다.
9. logger, 예외 처리, 반환 형식은 기존 코드 스타일과 맞춘다.
10. 기존 기능 회귀 테스트 항목을 반드시 포함한다.

---

## 새 기능 추가 시 필수 체크
새 기능을 추가할 때 반드시 아래를 확인한다.

- /tools 목록에서 새 기능이 보이는가?
- 기존 기능이 그대로 남아 있는가?
- 기존 액션 스키마와 충돌하지 않는가?
- 기존 health 응답이 유지되는가?
- GPT Actions 연결에 영향이 없는가?
- 기존 search_law_tool / get_law_detail_tool 동작이 그대로 유지되는가?

---

## 작업 출력 형식 규칙
코드 수정 요청을 받으면 항상 아래 순서로 답한다.

1. 수정/추가할 파일 목록 요약
2. 각 파일별 수정 이유
3. 실제 코드 전체
4. 기존 기능 영향 여부 체크리스트
5. 회귀 테스트 목록
6. Render 재배포 후 확인할 테스트 목록

---

## 테스트 규칙
새 기능 추가 후 반드시 아래 두 종류의 테스트를 모두 제시한다.

### 1. 기존 기능 회귀 테스트
- /health
- /tools
- search_law_tool
- list_law_names_tool
- get_law_detail_tool

### 2. 새 기능 테스트
- 새로 추가한 tool 목록 노출 여부
- 요청 파라미터 검증
- 정상 응답 여부
- 상세 조회 가능 여부
- 예외 처리 여부

---

## 배포 원칙
코드 수정 후 배포 절차도 항상 함께 안내한다.

기본 절차:
1. 코드 수정
2. 로컬 확인
3. GitHub commit / push
4. Render 자동 배포 확인
5. 자동 배포가 없으면 Deploy latest commit
6. /tools 확인
7. GPT Actions 스키마 반영
8. 새 채팅에서 실제 액션 테스트

주의:
- Restart service는 새 코드 반영용이 아니라는 점을 명시한다.
- 새 코드 반영은 commit/push 후 재배포가 필요하다.

---

## 스키마 원칙
OpenAPI 스키마는 서버에 실제로 존재하는 기능만 설명해야 한다.
서버에 없는 기능을 스키마에 먼저 추가하지 않는다.

순서:
1. 서버 코드에 기능 추가
2. 배포
3. /tools에서 실제 노출 확인
4. 그 다음 스키마 추가

---

## 응답 작성 스타일
- 항상 기존 코드 안정성을 먼저 고려한다.
- "기존 기능 보존"을 먼저 선언한다.
- 새 기능 구현 시 최소 변경 원칙을 유지한다.
- 추측성 변경을 피한다.
- 변경 범위를 명확히 설명한다.
- 사용자가 바로 적용할 수 있도록 실제 코드 중심으로 작성한다.

---

## 작업 모드
이 프로젝트에 대해 코드 수정 요청을 받으면 기본적으로 아래 전제를 따른다.

- 운영 중인 서버다
- 기존 기능이 이미 붙어 있다
- GPT Actions와 연결돼 있다
- 실수로 기존 기능이 깨지면 안 된다
- 안전한 점진 확장이 최우선이다

---

## 프로젝트 전용 기본 지시
이 프로젝트에서 작업할 때는 항상 다음을 우선 적용한다.

"기존 정상 기능을 절대 깨뜨리지 말고, 최소 변경으로 새 기능만 안전하게 추가하라."

"리팩토링보다 호환성과 안정성을 우선하라."

"새 기능을 만들 때는 /tools 노출, 예외 처리, 회귀 테스트, 재배포 절차까지 함께 제시하라."
