# LexGuard MCP 유지보수·확장 전용 Skill

## 목적
이 Skill은 LexGuard MCP 기반 한국 법령 검색 서버를 안전하게 유지보수하고 확장하기 위한 프로젝트 전용 작업 규칙이다.

이 프로젝트는 이미 운영 중인 서버이며, 기존 정상 기능을 절대 깨뜨리지 않는 것이 최우선이다.
새 기능 추가, 버그 수정, 엔드포인트 확장, GPT Actions 스키마 보강 시 반드시 이 문서를 따른다.

---

## 프로젝트 환경
- Render 배포 서버 사용
- GitHub 연동 배포 사용
- GPT Actions + OpenAPI 스키마 연결
- 현재 HTTP 기반 액션 환경 사용

---

## 현재 정상 동작 기능
다음 기능은 정상 동작 중인 것으로 간주한다.

- health
- search_law_tool
- list_law_names_tool
- get_law_detail_tool
- search_admin_rule_tool
- get_admin_rule_detail_tool

이 기능들의 엔드포인트, 파라미터명, 응답 구조는 함부로 변경하지 않는다.

---

## 최우선 원칙
1. 기존 정상 동작 기능 보존이 최우선이다.
2. 새 기능은 반드시 "추가 방식"으로 구현한다.
3. 기존 엔드포인트 경로를 변경하지 않는다.
4. 기존 도구 이름을 변경하지 않는다.
5. 기존 요청 파라미터명을 변경하지 않는다.
6. 기존 응답 키 구조를 함부로 변경하지 않는다.
7. health 기능은 특별한 이유 없으면 건드리지 않는다.
8. 대규모 리팩토링, 전면 구조 개편, 불필요한 파일 이동은 금지한다.
9. 운영 중인 서버라는 전제를 유지하고 최소 수정 원칙을 지킨다.

현재 GPT Actions와 연결되는 HTTP 도구는 아래를 기준으로 본다.

- health
- search_law_tool
- list_law_names_tool
- get_law_detail_tool

## 행정규칙 API 구현 참고 규격
행정규칙 기능을 수정하거나 확장할 때는 아래 공식 API 규격을 우선 기준으로 삼는다.

### 1. 행정규칙 목록 조회 API
- 요청 경로: `lawSearch.do`
- target: `admrul`

#### 주요 요청 파라미터
- `OC`: 필수
- `target=admrul`
- `query`: 검색어
- `page`: 페이지 번호
- `display`: 페이지당 결과 수
- `search`: 검색 범위
- `type`: 응답 형식

#### 목록 조회 결과에서 활용 가능한 값
- 행정규칙 일련번호
- 행정규칙 ID
- 행정규칙명
- 행정규칙 상세 링크
- 시행일자
- 소관부처명

### 2. 행정규칙 본문 조회 API
- 요청 경로: `lawService.do`
- target: `admrul`

#### 상세 조회 파라미터
- `ID`: 행정규칙 일련번호
- `LID`: 행정규칙 ID
- `LM`: 행정규칙명(정확한 이름)

#### 상세 조회 응답에 포함될 수 있는 내용
- 조문내용
- 부칙
- 별표
- 첨부파일 링크

---

## 행정규칙 상세 조회 구현 원칙
행정규칙 상세 조회는 이름 문자열 exact match에만 의존하지 않는다.

우선순위:
1. 목록 조회 결과에서 `ID` 또는 `LID`를 확보한다.
2. 상세 조회는 가능하면 `ID` 또는 `LID`를 우선 사용한다.
3. `LM`(행정규칙명)은 fallback 용도로만 사용한다.
4. 외부 인터페이스의 `rule_name` 파라미터는 유지하되, 내부적으로는 아래 순서로 처리한다.
   - exact match 시도
   - 실패 시 공백/괄호/특수문자 정규화
   - 실패 시 목록 조회 재검색
   - 가장 유력한 후보의 `ID` 또는 `LID`로 상세 조회
   - 끝까지 실패할 때만 기존 에러 반환
5. NFPC / NFTC / 기준번호 표기 차이로 실패하지 않도록 이름 정규화 로직을 둔다.

## 확장 원칙
새 기능을 추가할 경우 아래 순서를 따른다.

1. 기존 코드 구조를 먼저 읽고 이해한다.
2. 기존 service / repository / route 패턴을 재사용할 수 있는지 먼저 검토한다.
3. 새 request model이 필요하면 기존 모델 패턴에 맞춰 최소 추가한다.
4. service 계층은 기존 스타일대로 메서드만 확장한다.
5. route 계층은 기존 HTTP route 등록 방식에 맞춰 최소 분기만 추가한다.
6. `/tools` 목록에 새 도구가 실제로 노출되도록 추가한다.
7. 필수 파라미터 누락 시 기존과 동일하게 `error + recovery_guide` 형식으로 반환한다.
8. `page`, `per_page` float 입력은 기존 방식처럼 int 변환 처리한다.
9. logger, 예외 처리, 반환 형식은 기존 코드 스타일에 맞춘다.
10. 기존 기능 회귀 테스트를 반드시 포함한다.

## 작업 출력 형식 규칙
코드 수정 요청을 받으면 항상 아래 순서로 답한다.

1. 수정/추가할 파일 목록 요약
2. 각 파일별 수정 이유
3. 실제 코드 전체
4. 기존 기능 영향 여부 체크리스트
5. 회귀 테스트 목록
6. Render 재배포 후 확인할 테스트 목록

---

## 테스트 규칙
새 기능 추가 또는 수정 후 반드시 아래를 모두 확인한다.

### 기존 기능 회귀 테스트
- /health
- /tools
- search_law_tool
- list_law_names_tool
- get_law_detail_tool

### 행정규칙 기능 테스트
- search_admin_rule_tool
- get_admin_rule_detail_tool
- 검색 결과 이름을 그대로 넣은 상세 조회 성공 여부
- exact match 실패 시 fallback 로직 동작 여부

---

## 배포 원칙
코드 수정 후 배포 절차도 항상 함께 안내한다.

기본 절차:
1. 코드 수정
2. 로컬 확인
3. GitHub commit / push
4. Render 자동 배포 확인
5. 자동 배포가 없으면 Deploy latest commit
6. `/tools` 확인
7. GPT Actions 스키마 반영
8. 새 채팅에서 실제 액션 테스트

주의:
- Restart service는 새 코드 반영용이 아님
- 새 코드 반영은 commit/push 후 재배포가 필요함

---

## 스키마 원칙
OpenAPI 스키마는 서버에 실제로 존재하는 기능만 설명해야 한다.

순서:
1. 서버 코드에 기능 추가
2. 배포
3. `/tools`에서 실제 노출 확인
4. 그 다음 스키마 추가

---

## 프로젝트 전용 기본 지시
항상 아래를 우선 적용한다.

- 기존 정상 기능을 절대 깨뜨리지 말고, 최소 변경으로 새 기능만 안전하게 추가하라.
- 리팩토링보다 호환성과 안정성을 우선하라.
- 새 기능을 만들 때는 `/tools` 노출, 예외 처리, 회귀 테스트, 재배포 절차까지 함께 제시하라.
- 행정규칙 상세 조회는 이름 exact match보다 `ID/LID 우선 + 이름 fallback` 구조를 우선하라.
